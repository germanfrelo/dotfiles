# MANAGED.txt — Option 3: CI Verification

> **Purpose of this doc:** Full context and implementation spec for adding a CI check that verifies `MANAGED.txt` is up to date. Written for a fresh Copilot chat that has no prior conversation history.

---

## Background

`MANAGED.txt` is a human-readable tree of all chezmoi-managed dotfiles, committed at the repo root. It serves as documentation (shown in `README.md`) and as a quick audit surface.

It is auto-generated by `scripts/managed.js` and auto-staged by the Husky pre-commit hook (`.husky/pre-commit`). The output is a `tree --fromfile` rendering of all files in the git index under `home/` (after stripping chezmoi source-attribute prefixes such as `private_`, `dot_`, `run_once_before_`, etc.), plus an "Enforced absent" section for `remove_*` files.

### Repo layout

```text
dotfiles/                          ← git repo root
├── .husky/
│   └── pre-commit                 ← Husky hook: runs managed.js + lint-staged
├── home/                          ← chezmoi source root (.chezmoiroot = "home")
│   ├── .chezmoiscripts/           ← chezmoi run-scripts (included in MANAGED.txt)
│   ├── private_dot_config/        ← maps to ~/.config/
│   ├── private_Library/           ← maps to ~/Library/
│   └── …
├── scripts/
│   ├── managed.js                 ← generates MANAGED.txt
│   └── sync-chezmoi-config.js     ← injects chezmoi config data into docs
├── MANAGED.txt                    ← committed output; this file
└── package.json
```

### `package.json` scripts

```json
{
  "scripts": {
    "managed": "node scripts/managed.js > MANAGED.txt",
    "chezmoi:sync": "node scripts/sync-chezmoi-config.js",
    "prepare": "husky"
  },
  "lint-staged": {
    "*.*": ["prettier --write --ignore-unknown"]
  }
}
```

### `.husky/pre-commit`

```sh
# Regenerate MANAGED.txt only when staged changes affect the managed file list.
if git ls-files --cached --error-unmatch scripts/managed.js > /dev/null 2>&1; then
    if git diff --cached --name-only | grep -qE '^home/|^scripts/managed\.js$'; then
        node scripts/managed.js > MANAGED.txt
        git add MANAGED.txt
    fi
fi

npx lint-staged
```

The hook only regenerates `MANAGED.txt` when files under `home/` or `scripts/managed.js` itself are staged. It is intentionally NOT triggered by commits to docs, config, or other unrelated files.

### `scripts/managed.js` — current implementation (Option 1, already done)

```js
import { execFileSync, spawnSync } from "node:child_process";
import { readFileSync } from "node:fs";
import { join } from "node:path";

const repoRoot = new URL("../", import.meta.url).pathname;

// Reads from the git INDEX (staged + committed), not from disk.
const gitOutput = execFileSync("git", ["ls-files", "--cached", "home/"], {
  encoding: "utf8",
});

const allPaths = gitOutput.trim().split("\n").filter(Boolean);

const targetPaths = allPaths
  .filter((p) => !isChezmoiMetaFile(p) && !isRemoveFile(p))
  .map(sourcePathToTarget)
  .join("\n");

const result = spawnSync("tree", ["-a", "--fromfile", "."], {
  input: targetPaths,
  encoding: "utf8",
});
process.stdout.write(result.stdout);

const removePaths = allPaths.filter(isRemoveFile).sort();
if (removePaths.length > 0) {
  process.stdout.write("\n## Enforced absent\n\n");
  for (const sourcePath of removePaths) {
    const targetPath = sourcePathToTarget(sourcePath);
    const content = readFileSync(join(repoRoot, sourcePath), "utf8").trim();
    const adrRef = content.startsWith("adr:") ? content.slice(4).trim() : null;
    const adrSuffix = adrRef ? ` — [ADR](${adrRef})` : "";
    process.stdout.write(`- \`~/${targetPath}\`${adrSuffix}\n`);
  }
}

function isChezmoiMetaFile(path) {
  const first = path.slice("home/".length).split("/")[0];
  return first === ".chezmoi.toml.tmpl" || first === ".chezmoiignore";
}

function isRemoveFile(path) {
  const parts = path.slice("home/".length).split("/");
  return parts[parts.length - 1].startsWith("remove_");
}

function sourcePathToTarget(path) {
  const parts = path.slice("home/".length).split("/");
  return parts.map(chezmoiNameToTarget).join("/");
}

function chezmoiNameToTarget(name) {
  const attributePrefixes = [
    "private_",
    "readonly_",
    "executable_",
    "encrypted_",
    "exact_",
    "create_",
    "modify_",
    "symlink_",
    "run_",
    "once_",
    "onchange_",
    "before_",
    "after_",
    "remove_",
    "literal_",
    "empty_",
  ];
  let stripped = true;
  while (stripped) {
    stripped = false;
    for (const prefix of attributePrefixes) {
      if (name.startsWith(prefix)) {
        name = name.slice(prefix.length);
        stripped = true;
        break;
      }
    }
  }
  if (name.startsWith("dot_")) name = "." + name.slice(4);
  if (name.endsWith(".tmpl")) name = name.slice(0, -5);
  return name;
}
```

**Key design decision:** uses `git ls-files --cached home/` instead of the previous `chezmoi managed` call. `chezmoi managed` reads the chezmoi source directory on disk, which means any uncommitted file in `home/` pollutes the output. The git index approach makes MANAGED.txt immune to working-tree noise.

## Problem statement (why Option 3 is still needed)

The pre-commit hook only regenerates MANAGED.txt when `home/` files are staged. This means MANAGED.txt can become stale in two scenarios:

1. **Squashed merges / rebases**: Git may apply commits without running Husky.
2. **Direct edits to `MANAGED.txt`**: A manual edit could put it out of sync with reality.
3. **Future changes to `managed.js`** that alter the output format — if the hook does not trigger (no `home/` changes staged), MANAGED.txt silently drifts.
4. **CI bot commits** (e.g., from the `sync` workflow) that don't touch `home/` but could theoretically affect MANAGED.txt indirectly.

The pre-commit hook is a best-effort local guard. A CI check would be the authoritative, tamper-proof verification.

## Option 3: CI verification workflow

### Goal

Add a GitHub Actions job that runs `npm run managed` and verifies the output matches the committed `MANAGED.txt`. Fail the workflow (with a clear error) if they differ.

### When to run

Trigger on every push to `main` **and** on every PR. This catches drift regardless of whether Husky ran locally.

### Approach

Run `npm run managed` (which writes to `MANAGED.txt`) then check `git diff --exit-code MANAGED.txt`. If the diff is non-empty, the file is stale — fail with a helpful message.

### Implementation

#### 1. Create `.github/workflows/verify-managed.yml`

```yaml
name: Verify MANAGED.txt

on:
  push:
    branches: [main]
  pull_request:

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version-file: .nvmrc
          cache: npm

      - run: npm ci

      - name: Check MANAGED.txt is up to date
        run: |
          npm run managed
          if ! git diff --exit-code MANAGED.txt; then
            echo ""
            echo "MANAGED.txt is out of date. Run 'npm run managed' locally and commit the result."
            exit 1
          fi
```

**Notes:**

- `tree` must be available in the runner. On `ubuntu-latest`, `tree` is pre-installed. Verify with `which tree` in a test run if uncertain.
- `npm ci` installs dependencies (including `husky`). The `prepare` script runs `husky`, but Husky v9 skips installation in CI automatically when `CI=true` is set (GitHub Actions sets this). So Husky does NOT run in the CI environment — `npm run managed` calls the script directly without the hook.
- `node-version-file: .nvmrc` reads the `.nvmrc` at the repo root (currently `22` or similar — check the actual file).
- `npm run managed` runs `node scripts/managed.js > MANAGED.txt`, which uses `git ls-files --cached home/` — the git index is exactly what's checked out, so the output is deterministic.

#### 2. Optional: add a `.prettierignore` entry for MANAGED.txt (if not already present)

MANAGED.txt is auto-generated and not prettier-formatted. Check `.prettierignore`:

```text
# Already present (verify):
MANAGED.txt
```

If it is NOT in `.prettierignore`, add it. Otherwise lint-staged would reformat it and produce conflicts with the generated output.

#### 3. No changes needed to `scripts/managed.js` or `.husky/pre-commit`

The CI check is purely additive. The pre-commit hook remains as the local fast-path.

## Success criteria

- Pushing a commit where `npm run managed` produces different output than the committed `MANAGED.txt` causes the `Verify MANAGED.txt` workflow to fail.
- Pushing a commit where MANAGED.txt is accurate causes the workflow to pass.
- The workflow passes on `ubuntu-latest` without installing chezmoi (managed.js uses only `git` and `tree`, both pre-installed).

## Context: what was already implemented in the preceding session

### Option 1 (done)

`scripts/managed.js` was rewritten to replace `chezmoi managed --exclude=remove` with `git ls-files --cached home/`. This fixes the root cause of MANAGED.txt pollution from uncommitted working-tree files. The following functions were also updated/added:

- `chezmoiNameToTarget`: added missing run-script prefixes (`once_`, `onchange_`, `before_`, `after_`)
- `isChezmoiMetaFile`: filters `.chezmoi.toml.tmpl` and `.chezmoiignore` from the tree output
- `isRemoveFile`: detects `remove_*` files for the "Enforced absent" section
- `sourcePathToTarget`: replaces the old `chezmoiSourceToTargetPath` helper

### CI bug fix (done)

`scripts/sync-chezmoi-config.js` `injectSection` function was updated to compare new content with existing content before calling `writeFileSync`. When content is unchanged, the file is NOT written (no mtime change, no false git modification). This fixes the `sync` workflow's "Prevented an empty git commit!" failure caused by lint-staged receiving an empty staged diff.
